# 1.12.2 Forge 数据采集 Mod 设计讨论稿

> 当前文档只讨论前两个阶段。  
> 第一阶段：基础轨迹采集与里程累加。  
> 第二阶段：出口设置工具，用 WorldEdit 木斧式两点选区定义出口区域，并在录制样本中标注玩家当前所在区域 ID。

- 停止录制后导出 JSON 到 `nebulaestats/yyyy-MM-dd_HH-mm-ss.json`。
- 出口设置工具：左键设置 `pos1`，右键设置 `pos2`。
- 出口设置工具：选区完整后，右键空气打开出口区域管理 GUI。
- 出口区域命令：`/ncs exit create/remove/list/info/gui`。
- 出口 ID 支持中文等 Unicode 字母/数字，以及 `_`、`-`、`:`、`.`。
- 出口区域持久化：保存到 `nebulaestats/exit_regions.json`，重启游戏后加载。

- 出口区域管理 GUI：选区完整后右键空气打开，也可用 `/ncs exit gui` 打开；支持添加、删除、滚动列表和响应式布局。

- 修改已有出口区域边界。
- 维度记录与跨维度区域管理。

## 1. 第一阶段目标

第一阶段先做一个稳定的“轨迹与里程采集器”。

每 tick 做这些事：

1. 保存玩家当前位置 `x/y/z`。
2. 保存玩家水平朝向 `yaw`。
3. 保存当前样本时间 `timeSeconds`，时间从 Java 标准库读取操作系统时间。
4. 用当前 tick 与上一 tick 的位置差计算本 tick 移动距离。
5. 将每 tick 移动距离向后累加，得到类似汽车里程表的累计里程。

核心模型：

```text
tick 0: 记录 P0，里程 = 0
tick 1: 记录 P1，距离 += distance(P0, P1)
tick 2: 记录 P2，距离 += distance(P1, P2)
tick 3: 记录 P3，距离 += distance(P2, P3)
...
```

这个里程不是“到某个目标的距离”，而是玩家实际移动过的累计距离。

## 2. 每 tick 采集的数据

### 2.1 玩家位置

每 tick 单独保存玩家坐标：

```json
{
  "x": 1000.5,
  "y": 64.0,
  "z": -500.5
}
```

建议保留 double 精度，不要只存整数方块坐标。

### 2.2 水平朝向

只记录玩家水平朝向：

```json
{
  "yaw": 90.0
}
```

`yaw` 用于表示玩家或镜头面向的水平角度。

### 2.3 时间

每个样本保存：

```json
{
  "tick": 12345,
  "timeSeconds": 617.25
}
```

`tick` 是本次录制内的采样序号或游戏 tick 计数。

`timeSeconds` 不使用 `tick / 20.0` 推算，而是使用 Java 标准库从操作系统时间获取。建议在录制开始时保存起始时间：

```text
recordStartNanos = System.nanoTime()
```

每个样本的时间：

```text
timeSeconds = (System.nanoTime() - recordStartNanos) / 1_000_000_000.0
```

这样即使游戏 TPS 波动，样本时间也能反映真实经过时间。

## 3. 里程累加算法

### 3.1 三维里程

本 tick 移动距离：

```text
dx = current.x - previous.x
dy = current.y - previous.y
dz = current.z - previous.z

deltaDistance3d = sqrt(dx * dx + dy * dy + dz * dz)
```

累计三维里程：

```text
totalDistance3d += deltaDistance3d
```

三维里程会把跳跃、上坡、下坡、飞行的高度变化都算进去。

### 3.2 水平里程

如果想更像汽车里程表，通常应该只计算水平移动：

```text
dx = current.x - previous.x
dz = current.z - previous.z

deltaDistance2d = sqrt(dx * dx + dz * dz)
```

累计水平里程：

```text
totalDistance2d += deltaDistance2d
```

第一阶段建议同时保存：

- `totalDistance2d`：水平里程，更像汽车里程。
- `totalDistance3d`：三维轨迹长度，更完整。

后续显示或导出时再决定用哪个。

## 4. 单个数据样本结构

每 tick 的样本结构建议如下：

```json
{
  "tick": 12500,
  "timeSeconds": 625.0,
  "position": {
    "x": 1100.0,
    "y": 64.0,
    "z": -700.0
  },
  "yaw": 90.0,
  "delta": {
    "dx": 0.12,
    "dy": 0.0,
    "dz": 0.05,
    "distance2d": 0.13,
    "distance3d": 0.13
  },
  "odometer": {
    "distance2d": 1685.5,
    "distance3d": 1692.8
  }
}
```

这里的 `odometer.distance2d` 就是“汽车里程表式”的累计里程。

第一阶段不记录维度，也不考虑传送。

## 5. 录制文件结构

建议输出一个 JSON 文件：

```text
nebulaestats/yyyy-mm-dd_hh-mm-ss.json
```

文件结构：

```json
{
  "meta": {
    "minecraftVersion": "1.12.2",
    "sampleRate": "1 tick",
    "timeSource": "System.nanoTime"
  },
  "summary": {
    "sampleCount": 12000,
    "durationSeconds": 600.0,
    "totalDistance2d": 1685.5,
    "totalDistance3d": 1692.8
  },
  "samples": []
}
```

`samples` 中保存每 tick 的位置、yaw、delta 和累计里程。

## 7. 第一阶段实现目标

### 7.1 必须实现

1. 每 tick 读取玩家 `x/y/z`。
2. 每 tick 读取玩家水平朝向 `yaw`。
3. 每 tick 使用 Java 标准库从操作系统时间计算 `timeSeconds`。
4. 保存每 tick 原始位置。
5. 计算当前 tick 与上一 tick 的 `dx/dy/dz`。
6. 计算 `deltaDistance2d`。
7. 计算 `deltaDistance3d`。
8. 累加 `totalDistance2d`。
9. 累加 `totalDistance3d`。
10. 停止录制后保存 JSON。
11. JSON 中每个样本都带有 tick、timeSeconds、位置、yaw、单 tick 位移、累计里程。

### 7.2 第一阶段未纳入范围

- 不记录维度。
- 不考虑传送。
- 不计算到终点的距离。
- 不计算到出口的距离。
- 不做路径插值。
- 不做 AE 导入。
- 不生成 AE 脚本。
- 不和其他 Minecraft Mod 联动。

### 7.3 第一阶段验收例子

```text
1. 玩家执行 /ncs start。
2. 玩家沿道路移动一段距离。
3. Mod 每 tick 保存 XYZ、yaw 和从操作系统时间计算出的 timeSeconds。
4. Mod 每 tick 根据前后位置差累加里程。
5. 玩家执行 /ncs stop。
6. 生成 JSON 文件。
7. JSON summary 中可以看到 totalDistance2d 和 totalDistance3d。
8. samples 中可以看到每 tick 的位置、yaw、timeSeconds 和累计里程。
```

## 8. 关键设计注意点

### 8.1 是否计算 Y 轴

建议同时记录：

- 水平里程 `distance2d`
- 三维里程 `distance3d`

如果目标是汽车导航风格，优先用 `distance2d`。如果目标是完整运动轨迹，使用 `distance3d`。

### 8.2 玩家静止时

如果玩家没有移动：

```text
dx = 0
dy = 0
dz = 0
deltaDistance = 0
totalDistance 不变
```

仍然继续记录 yaw 和 timeSeconds，因为玩家可能原地转动视角，时间也仍然在推进。

## 9. 第二阶段实现目标：出口设置工具

第二阶段的目标是做“出口设置工具”。

这个工具不是“到达某个点后右键保存当前点”，而是类似 WorldEdit 木斧：

**玩家手持工具选择两个点，由这两个点确定一片立体区域。确定后用指令给这片区域设置 ID。录制过程中，每 tick 记录数据样本时，如果玩家当前位置处于某个出口区域内，就在该样本中标注这个区域 ID。**

### 9.1 工具行为

建议新增一个物品：

```text
出口设置工具 / Exit Settings Tool
```

操作方式：

```text
左键方块：选择 pos1
右键方块：选择 pos2
右键空气：如果 pos1/pos2 已完成，则打开出口区域管理 GUI
```

`pos1` 和 `pos2` 确定一个 AABB 立体区域。

然后使用指令为这片区域设置 ID：

```text
/ncs exit create <id>
```

示例：

```text
/ncs exit create exit_1685
```

### 9.2 出口区域结构

出口区域保存为：

```json
{
  "id": "exit_1685",
  "min": {
    "x": 1150,
    "y": 50,
    "z": -850
  },
  "max": {
    "x": 1250,
    "y": 90,
    "z": -750
  }
}
```

第一版只需要 AABB，不需要多边形、圆形、路径投影或复杂区域。

### 9.3 样本中的区域标注

第二阶段在每 tick 记录样本时，额外判断玩家当前位置是否处于某个出口区域内。

判断逻辑：

```text
inside =
  player.x >= area.min.x && player.x <= area.max.x &&
  player.y >= area.min.y && player.y <= area.max.y &&
  player.z >= area.min.z && player.z <= area.max.z
```

如果玩家在区域内，则样本中标注区域 ID：

```json
{
  "tick": 12500,
  "timeSeconds": 625.0,
  "position": {
    "x": 1200.0,
    "y": 64.0,
    "z": -800.0
  },
  "yaw": 90.0,
  "exitRegionId": "exit_1685",
  "delta": {
    "dx": 0.12,
    "dy": 0.0,
    "dz": 0.05,
    "distance2d": 0.13,
    "distance3d": 0.13
  },
  "odometer": {
    "distance2d": 1685.5,
    "distance3d": 1692.8
  }
}
```

如果玩家不在任何出口区域内：

```json
{
  "tick": 12501,
  "timeSeconds": 625.05,
  "position": {
    "x": 1201.0,
    "y": 64.0,
    "z": -801.0
  },
  "yaw": 90.0,
  "exitRegionId": null
}
```

如果后续允许区域重叠，可以改成：

```json
{
  "exitRegionIds": ["exit_1685", "exit_ramp_a"]
}
```

但第二阶段先用单个 `exitRegionId` 就够。

### 9.4 JSON 文件扩展

第二阶段在 JSON 中增加 `exitRegions` 字段，并让样本支持 `exitRegionId`。

```json
{
  "exitRegions": [
    {
      "id": "exit_1685",
      "min": { "x": 1150, "y": 50, "z": -850 },
      "max": { "x": 1250, "y": 90, "z": -750 }
    }
  ],
  "samples": [
    {
      "tick": 12500,
      "timeSeconds": 625.0,
      "position": { "x": 1200.0, "y": 64.0, "z": -800.0 },
      "yaw": 90.0,
      "exitRegionId": "exit_1685",
      "odometer": {
        "distance2d": 1685.5,
        "distance3d": 1692.8
      }
    }
  ]
}
```

当前代码还会把出口区域独立持久化到：

```text
nebulaestats/exit_regions.json
```

服务端启动时加载该文件，服务端停止时保存；通过命令或 GUI 创建/删除区域后也会立即保存。

### 9.5 第二阶段命令建议

```text
/ncs exit create <id>
/ncs exit remove <id>
/ncs exit list
/ncs exit info <id>
/ncs exit gui
```

其中 `create` 使用当前工具选中的 `pos1/pos2`。

当前代码还实现了出口区域管理 GUI：

- 选区完整后，手持工具右键空气打开 GUI。

- GUI 的创建/传送/删除操作通过 packet 交给服务端执行。

### 9.7 后续改进：区域可视化与传送辅助

第二阶段基础功能稳定后，已经继续增加两个编辑辅助功能。

#### 9.7.1 手持工具时显示出口区域边界

当玩家手持“出口设置工具”时，在客户端显示已有出口区域的空间边界，效果参考原版手持屏障方块时显示屏障粒子的体验。

当前实现使用线框显示 AABB 边界，而不是持续生成真实粒子。这样边界更清晰，也不会向世界里写入额外效果。

建议行为：

- 只有玩家手持出口设置工具时显示，避免平时干扰视野。
- 显示当前世界内已定义出口区域的 AABB 边界。
- 使用半透明线框表达区域范围。
- 优先显示玩家附近一定距离内的区域，避免大量区域造成性能压力。
- 当前正在选择的 `pos1/pos2` 区域可以用不同颜色或更高亮的粒子显示。
- 玩家准星指向某个出口区域时，显示类似实体命名牌的出口名提示框。
- 显示逻辑只在客户端渲染，不影响服务端区域判定和录制数据。

实现方向：

```text
client tick / render world last
  -> 判断玩家是否手持 nebulaestats:exit_settings_tool
  -> 从客户端缓存的出口区域列表读取 min/max
  -> 在区域边界或角点生成可视化粒子/线框
```

#### 9.7.2 GUI 增加传送按钮

出口区域管理 GUI 的每个出口条目右侧，已经在“删除”按钮左侧增加一个“传送”按钮。

按钮行为：

- 点击“传送”后，请求服务端把玩家传送到该出口区域。
- 默认传送坐标使用区域 `min` 位置，并把 Y 坐标加 1：

```text
teleportX = region.min.x + 0.5
teleportY = region.min.y + 1
teleportZ = region.min.z + 0.5
```

说明：

- `+0.5` 用于让玩家落在方块中心。
- `y + 1` 用于避免玩家传送到区域底部方块内部。
- 服务端必须重新校验区域是否存在，不能只信任客户端传来的坐标。
- 如果后续支持维度，传送时还需要校验玩家所在维度与区域所属维度。

## 10. 当前建议结论

第一阶段定义为：

**每 tick 保存玩家 XYZ、yaw、系统时间 timeSeconds，并通过相邻 tick 位移累加得到里程表数据。**

第二阶段定义为：

**新增出口设置工具。工具像 WorldEdit 木斧一样用两点定义出口区域，指令设置区域 ID。录制样本时，如果玩家位置处于该区域内，就在样本中写入 `exitRegionId`。**

这样得到的数据基础很干净：

- 每 tick 原始位置。
- 每 tick yaw。
- 每 tick 系统时间。
- 每 tick 位移。
- 累计水平里程。
- 累计三维里程。
- 第二阶段开始，每 tick 可选出口区域 ID。
