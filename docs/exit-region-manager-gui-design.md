# 出口区域管理 GUI 设计

## 1. 目标

为出口区域管理增加一个方便的游戏内 GUI。

这个 GUI 的目标是减少路线标记过程中的命令输入：

- 玩家选择完 `pos1` 和 `pos2` 后，手持工具右键空气打开管理窗口。
- 玩家可以在顶部输入框里输入出口区域名称。
- 玩家可以把当前两点选区创建为出口区域。
- 窗口下方列出现有出口区域。
- 每个已有出口区域右侧都有传送和删除按钮。

当前代码状态：

- 已实现：选区完整后右键空气打开 GUI、`/ncs exit gui`、添加、传送、删除、滚动列表、响应式布局、服务端同步与持久化保存、手持工具时区域线框预览、选区变化实时同步。
- 未实现：重命名、修改边界、搜索/筛选。

视觉方向参考原版“社交”窗口：

- 背景世界被压暗，必要时可做轻微模糊感。
- 中央灰色面板。
- 顶部黑色输入框。
- 中间/下方是可滚动列表。
- 使用原版风格矩形按钮。

## 2. 当前数据模型

现有类：

- `ExitRegion`
  - `id`
  - `min`
  - `max`
- `ExitRegionManager`
  - 添加、删除、列出、查找区域。
- `ExitRegionStorage`
  - 从 `nebulaestats/exit_regions.json` 保存/读取区域。
- `SelectionManager`
  - 按玩家保存 `pos1` 和 `pos2`。

当前命令流程：

```text
手持工具左键方块  -> 设置 pos1
手持工具右键方块  -> 设置 pos2
/ncs exit create <id>
/ncs exit remove <id>
/ncs exit list
/ncs exit info <id>
```

GUI 应该建立在这个模型之上，而不是替换它。命令仍然保留，方便调试和备用操作。

## 3. 用户流程

### 3.1 确认后打开

```text
1. 玩家手持出口设置工具。
2. 左键方块设置 pos1。
3. 右键方块设置 pos2。
4. 玩家确认选区无误后，右键空气打开“出口区域管理”GUI。
5. 玩家输入 ID，例如：上虞。
6. 玩家点击“添加”。
7. 服务端创建区域并保存。
8. GUI 刷新区域列表。
```

### 3.2 手动打开

增加命令：

```text
/ncs exit gui
```

后续可选快捷键：

```text
F10 -> 打开出口区域管理窗口
```

## 4. GUI 布局

建议布局：

```text
+--------------------------------------------------+
|                 出口区域管理                     |
|                                                  |
|  出口名: [ 上虞________________________ ] [添加] |
|                                                  |
|  当前选区                                        |
|  pos1: x y z                                     |
|  pos2: x y z                                     |
|  size: w x h x d                                 |
|                                                  |
|  已定义出口                                      |
|  +--------------------------------------------+  |
|  | 上虞             min/max 坐标     [删除]    |  |
|  +--------------------------------------------+  |
|  | 出口_一号        min/max 坐标     [删除]    |  |
|  +--------------------------------------------+  |
+--------------------------------------------------+
```

### 4.1 顶部输入框

使用 `GuiTextField`。

行为：

- 支持中文输入，前提是 Minecraft 当前输入法本身能正常输入中文。
- 使用和命令一致的 ID 校验规则：
  - Unicode 字母。
  - Unicode 数字。
  - `_`
  - `-`
  - `:`
  - `.`
- 输入为空时禁用“添加”按钮。
- 如果 `pos1/pos2` 不完整，也禁用“添加”按钮。

### 4.2 当前选区区域

显示：

- `pos1`
- `pos2`
- 归一化后的 `min/max`
- 区域尺寸

如果选区不完整，显示：

```text
请先用出口设置工具选择两个点
```

### 4.3 出口区域列表

每一行显示：

- 区域 ID。
- `min` 坐标。
- `max` 坐标。
- 传送按钮。
- 删除按钮。

当前实现支持鼠标滚轮滚动、右侧滚动条拖动和列表区域裁剪；窗口较小时显示较少行，窗口较大时最多显示 5 行。

## 5. 客户端/服务端职责

关键规则：

GUI 是客户端界面，但出口区域数据必须由服务端负责。

原因：

- 录制逻辑在服务端执行。
- 玩家是否进入出口区域的判断在服务端执行。
- 持久化保存也在服务端执行。
- 如果以后进入多人环境，必须以服务端为准。

因此 GUI 按钮不能直接修改客户端数据，而应该发送 packet 给服务端处理。

## 6. 网络同步方案

使用 Forge `SimpleNetworkWrapper`。

建议 channel：

```text
nebulaestats
```

当前已实现 packet：

### 6.1 `PacketOpenExitRegionGui`

方向：

```text
server -> client
```

用途：

- 在玩家右键空气确认后通知客户端打开 GUI。
- 可以携带当前选区和当前区域列表。

### 6.2 `PacketSyncExitRegions`

方向：

```text
server -> client
```

用途：

- 把服务端当前所有出口区域同步给客户端。
- 打开 GUI 时使用。
- 创建/删除区域后使用。

载荷示例：

```json
[
  {
    "id": "上虞",
    "min": { "x": 1, "y": 64, "z": 2 },
    "max": { "x": 5, "y": 70, "z": 8 }
  }
]
```

实现上可以用 `ByteBufUtils.writeUTF8String` 序列化 JSON 字符串。

### 6.3 `PacketCreateExitRegion`

方向：

```text
client -> server
```

载荷：

```text
id
```

服务端应使用发送者玩家当前的服务端选区。

服务端行为：

```text
校验 id
检查 pos1/pos2
创建 ExitRegion
保存 exit_regions.json
把更新后的区域列表同步回玩家
```

### 6.4 `PacketDeleteExitRegion`

方向：

```text
client -> server
```

载荷：

```text
id
```

服务端行为：

```text
删除区域
保存 exit_regions.json
把更新后的区域列表同步回玩家
```

### 6.5 `PacketTeleportToExitRegion`

方向：
```text
client -> server
```

载荷：
```text
id
```

用途：

- GUI 中每个出口条目在“删除”按钮左侧增加“传送”按钮。
- 点击后请求服务端把玩家传送到对应出口区域。

服务端行为：

```text
根据 id 查找 ExitRegion
如果不存在，给玩家发送错误提示
如果存在，传送到：
  x = min.x + 0.5
  y = min.y + 1
  z = min.z + 0.5
```

注意：

- 传送必须由服务端执行。
- 客户端只发送区域 ID，不直接发送坐标。
- 后续如果出口区域记录维度，传送前必须校验或切换维度。

## 7. 第一版实现范围

当前已实现：

- `/ncs exit gui`

- “添加”按钮。
- 可滚动出口列表。

- 通过 packet 让服务端创建/传送/删除区域。

- 继续使用现有 `ExitRegionStorage` 做持久化保存。

- 支持列表区域裁剪，避免小窗口下内容溢出。
- 支持手持出口设置工具时显示附近出口区域线框。

- 区域导入/导出。

## 7.1 后续版本建议

第一版 GUI 稳定后，已经优先补上两个编辑效率功能。

### 7.1.1 手持工具显示区域范围

当玩家手持出口设置工具时，在客户端显示已定义出口区域的空间范围，体验参考原版手持屏障方块时显示屏障粒子的效果。

当前实现使用客户端线框渲染 AABB，不生成真实粒子。已有出口使用绿色线框，当前选区使用更醒目的蓝色线框。
设置 `pos1` 或 `pos2` 后，服务端会立即把当前选区同步到客户端，因此线框不需要等打开 GUI 后才刷新。
当玩家准星指向某个出口区域时，会显示类似实体命名牌的出口名提示框。

建议：

- 仅手持 `nebulaestats:exit_settings_tool` 时显示。
- 显示玩家附近的出口区域，避免所有区域同时渲染导致画面混乱。
- 使用粒子点、角点标记、边线粒子或半透明线框表现 AABB。
- 当前正在选择的 `pos1/pos2` 可以使用更醒目的颜色。
- 可视化仅作为编辑辅助，不参与实际区域判定。

### 7.1.2 条目增加传送按钮

出口列表每一行从右到左建议为：

```text
[出口名] [min/max 坐标] [传送] [删除]
```

传送按钮默认把玩家传送到该区域的 `min` 位置上方：

```text
x = min.x + 0.5
y = min.y + 1
z = min.z + 0.5
```

这样方便玩家快速回到某个出口区域进行复查、补录或重新选区。

## 8. 视觉说明

使用接近原版的风格：

- 深色半透明背景。
- 灰色面板边框。
- 黑色输入框。
- 列表行使用交替深灰。
- 白色文字。
- 删除按钮可以使用深红/暗色强调。
- 坐标标签保持紧凑。

避免：

- 过于现代的扁平 UI。
- 大型装饰图标。
- 不像 Minecraft 的渐变。
- 过大的文字。

## 9. 边界情况

### 9.1 ID 重复

服务端拒绝，并发送聊天提示：

```text
Exit region already exists: 上虞
```

后续改进：

- 在 GUI 内部用红字提示。

### 9.2 ID 非法

服务端拒绝：

```text
Exit region id may only contain Unicode letters, numbers, _, -, :, and .
```

### 9.3 选区缺失

客户端禁用“添加”按钮。

服务端也必须校验并拒绝不完整选区。

### 9.4 录制中删除区域

第一版允许。

原因：

- 每个 sample 会使用采样时刻的区域状态。
- 停止录制导出 JSON 时，会写入当时的 `exitRegions`。

后续可以考虑：

- 录制过程中锁定区域编辑。
